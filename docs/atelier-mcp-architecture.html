The network

Three nodes, two edges.

The agent never talks to a tool directly. It asks the Gateway "what's available?", gets back a list of tool descriptions, and invokes the ones it needs through the same channel. The Gateway is the seam.

Network · this session
3 nodes · 2 edges
AThe agent.
Owned

Strands orchestrator (Haiku) plus five specialists (Opus). On startup, fetches the tool catalog from the Gateway. On every turn, decides which tools to invoke and routes calls back through the Gateway.

BThe Gateway.
Managed

AgentCore Gateway. Publishes the tool surface as MCP, handles authentication, observability, and rate limiting. We don't run the protocol layer ourselves — Gateway does it. We could swap in our own MCP server; we don't need to.

CThe tools.
Owned

Our seven @tool-decorated functions. Registered with the Gateway at boot. The Gateway exposes their signatures (typed inputs, typed outputs, descriptions) — which is what the agent uses to choose between them.

The two edges
A → B Discovery. Agent asks the Gateway for the tool catalog. Returns names, signatures, descriptions. gateway.list_tools()
A → B Invocation. Agent calls a tool by name, Gateway routes to the implementation, returns the result. gateway.invoke("search_products", args)
B ⇢ C Implementation routing. Hidden from the agent. Could be in-process, could be remote. The agent doesn't care.
