The journey

Step by step, handoff by handoff.

Each step shows what the layer received and what it passed forward. Following the right column tells you the data shape at every seam — the truth of the runtime is the handoffs, not the implementations.

i.

Triage fastpath.

deterministic · 20ms

The request lands at POST /api/chat/stream. Before anything LLM fires, a regex pass checks for greetings, thanks, and meta-queries. If matched, a canned reply ships and the turn ends here. No agent invocation.

Handoff
→in: message · session
→out: match? → canned20ms
ii.

Intent classification.

deterministic · 80ms

A small classifier (rules + small embedding) tags the message as recommend, search, support, pricing, or inventory. The tag prepends an [USE: …] hint so the orchestrator can route deterministically rather than reasoning about routing.

Handoff
→in: message
→out: intent + hint78ms
iii.

Skill routing.

Haiku · 280ms

One short Haiku call against the skill registry's descriptions. Returns a JSON list of skill names to load. The loaded skills are injected into the specialist's system prompt via a ContextVar — the orchestrator itself doesn't carry skills.

Handoff
→in: message · skills.list()
→out: RouterDecision287ms
iv.

Orchestrator.

Haiku · 420ms

The dispatcher. Reads the intent hint, picks one of the five specialists, and hands off. Doesn't reason about the answer — that's the specialist's job. Cheap and fast on purpose.

Handoff
→in: hint + msg + STM
→out: specialist.invoke()418ms
v.

Specialist agent.

Opus · 1,800ms

Opus reasoning, with the loaded skills appended to the base system prompt. Reads STM, may pull LTM, decides which tools to fire, composes the streaming reply. Where the budget mostly goes — and where the editorial voice gets shaped.

Handoff
→in: prompt + skills + ctx
→out: SSE token stream~1.8s
vi.

Tools, through the gateway.

parallel · 280ms

During step v, the specialist invokes one or more tools — usually search_products, check_inventory, get_recommendations. The Gateway routes calls; results return to the specialist mid-reasoning. Cost is wall-clock parallel with v.

Handoff
→in: tool name + args
→out: typed result42–412ms
vii.

Stream close, session write.

sync · 80ms async · LTM

The SSE stream emits its complete event. Synchronously: write the new turn back to session_state. Asynchronously, in a background job: extract durable facts and upsert them into LTM via Aurora pgvector. The next turn starts hot.

Handoff
→sync: session.upsert()8ms
→async: ltm.extract()background
When in doubt

Three rules about latency.

FAST FIRST
Cheap layers up front.
"Why classify before reasoning?"
  • Fastpath catches greetings · ~20ms
  • Intent classifier routes deterministically · ~80ms
  • Each cheap layer that succeeds saves 1.8s of Opus
  • Don't make the smart model decide what the small model can
PARALLEL
Tools while reasoning.
"Why don't tool calls stack?"
  • Specialist invokes tools mid-stream
  • Multiple tools fire concurrently when independent
  • Wall-clock cost = max(tool latencies), not sum
  • Sequential tool chains are a smell
DEFER
Async what can wait.
"What blocks the user vs runs after?"
  • Reply streaming blocks · everything else doesn't
  • LTM extraction · async
  • Telemetry · async
  • Time-to-first-token is the only number the user feels
